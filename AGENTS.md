## Package

`df_router` — a lightweight Flutter router published to pub.dev. Built on the `dev-cetera` package family (`df_pod` for reactive state, `df_safer_dart` for Result/Option types, `df_pwa_utils` for browser URL integration, `df_log` for logging). All four are sibling packages and are wired via `pubspec_overrides.yaml` to `../df_pod`, `../df_safer_dart`, `../df_log`, `../df_pwa_utils` — changes there are picked up locally without a pub publish.

## Common commands

```zsh
flutter pub get
flutter analyze                      # uses analysis_options.yaml (strict-casts/inference/raw-types + custom_lint)
dart run custom_lint                 # runs df_safer_dart_lints
dart format .
dart fix --apply
cd example/hosted_example && flutter run -d chrome   # the published live demo (deployed via .github/workflows/deploy-example.yml)
cd example/simple_example && flutter run -d chrome   # smaller standalone demo
```

There is no test suite — `flutter test` will be empty. The `example/example.dart` file is a single-file pub.dev example, not a runnable app.

## Release workflow

Releases are commit-message-driven through `.github/workflows/process-package.yml`:

- Commits starting with `+` → CI runs `dart format`, `dart fix --apply`, rewrites `CHANGELOG.md` and `README.md` from a central template, bumps the version, and pushes a `v{version}` git tag.
- Commits starting with `++` → same as above, plus publishes to pub.dev.
- `pubspec.yaml`'s `version:` is the source of truth — bump it in the same commit as the `+`/`++` message.
- `README.md` is **regenerated by CI** from `_README_CONTENT.md` + an external template. Edit `_README_CONTENT.md` (between the `<!-- BEGIN _README_CONTENT -->` / `<!-- END _README_CONTENT -->` markers in `README.md`) — direct README edits will be overwritten on the next release commit.

## Architecture

The router is built around a small set of cooperating types. Understanding the relationship between them is the fastest way to be productive.

### Core flow

1. **`RouteManager`** (`lib/src/route_manager.dart`) is the root widget. It instantiates a single `RouteController` and exposes it via `RouteControllerProvider` (an `InheritedWidget`). `StatelessRouteManager` is a variant that rebuilds the controller on every parent rebuild — prefer `RouteManager` unless you specifically need that.
2. **`RouteController`** (`lib/src/route_controller.dart`) owns navigation state in a `Pod<_NavigationState>` (list of routes + current index — a forward/backward history, not a stack-of-pushes). It listens to `platformNavigator` (from `df_pwa_utils`) for browser URL changes and calls `platformNavigator.pushState` to keep the URL in sync.
3. **`RouteState<TExtra>`** (`lib/src/route_state.dart`) is an `Equatable` value type wrapping a `Uri` plus typed `extra`, an `AnimationEffect`, a `condition`, and `shouldPreserve`. Equality is `(uri, extra)` — two `RouteState`s with the same path+query+extra are the same cache key. Subclass it per route in your app.
4. **`RouteBuilder<TExtra>`** (`lib/src/route_builder.dart`) pairs a `RouteState` template with a widget builder and per-route flags `shouldPreserve` (keep widget in memory after navigating away) and `shouldPrebuild` (build eagerly on controller init even if not the active route).
5. **`RouteWidgetMixin<TExtra>`** (`lib/src/route_widget_mixin.dart`) — screen widgets mix this in to expose `routeState` to the controller.

### How navigation actually renders

This is the non-obvious part — the router does **not** swap widgets on navigation. Instead:

- `RouteController` maintains a `_widgetCache: Map<RouteState, Widget>`. Every route the user has visited (or that's `shouldPrebuild`) lives in this map simultaneously.
- `buildScreen` renders **all** cached widgets inside a `PrioritizedIndexedStack` (`lib/src/prioritized_indexed_stack/`), with the current route on top and the previous route directly behind it for the duration of the transition animation.
- Animations are driven by `AnimationEffectBuilder` + `AnimationEffect` subclasses in `lib/src/prioritized_indexed_stack/effects.dart` (Cupertino, Material, Fade, SlideUp/Down, Forward/Backward, PageFlapDown…). Each effect returns per-layer transforms/opacity given a `0.0→1.0` animation value.
- When a transition completes, `_maybeRemoveStaleRoute` consults the **preservation strategy** (`_TPreservationStrategy`). If the route should not be preserved, its widget slot is replaced with a `SizedBox.shrink` (kept as a placeholder so the indexed stack indices stay stable) — it is not removed from the map.
- Customize cache lifecycle via `controller.setPreservationStrategy(...)`. The default ORs `RouteBuilder.shouldPreserve` and `RouteState.shouldPreserve`.

This design is what enables the "stateful routes" feature: a `shouldPreserve: true` screen keeps its widget instance (and therefore its `State`) across visits.

### Routing matches on `uri.path` only

`_builderMap` keys are `routeState.uri.path`, so `MessagesRouteState1` (`/messages?key1=value1`) and `MessagesRouteState2` (`/messages?key2=value2`) resolve to **the same `RouteBuilder`** but produce **different cache keys** (because `RouteState` equality includes the full uri). Query parameters and `extra` differentiate cache entries but not builders. When pushing a `Uri` that has no registered builder for its path, the controller logs an error and (if provided) navigates to `errorRouteState`.

### Public API surface

`lib/df_router.dart` exports `lib/src/_src.g.dart` (a **generated** barrel from `df_generate_dart_indexes` — do not edit by hand; add new files and regenerate) and re-exports all of `df_pwa_utils`. Consumers import `package:df_router/df_router.dart` and get everything.

### `MaterialApp` integration gotcha

Mount `RouteManager` inside `MaterialApp.builder`, not `MaterialApp.home` — `home` conflicts with the router's screen swapping. The README and `example/hosted_example/lib/main.dart` demonstrate the correct pattern.

## Style rules from `analysis_options.yaml`

Strict mode is on (`strict-casts`, `strict-inference`, `strict-raw-types`) and many lints are promoted to errors (`prefer_relative_imports`, `flutter_style_todos`, `unnecessary_new`, `use_key_in_widget_constructors`, `no_leading_underscores_for_local_identifiers`, …). `custom_lint` runs `df_safer_dart_lints` on top. Use single quotes, trailing commas, relative imports within the package.
